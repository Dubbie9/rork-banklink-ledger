# Clean Architecture Migration Guide\n\nThis document outlines the refactored clean architecture structure and how to migrate existing code.\n\n## Architecture Overview\n\nThe codebase has been refactored following Clean Architecture principles with clear separation of concerns:\n\n```\nsrc/\n├── core/                     # Core business logic & domain\n│   ├── entities/            # Domain entities (User, Bank, Transaction)\n│   ├── use-cases/           # Business logic use cases\n│   └── repositories/        # Repository interfaces\n├── infrastructure/          # External concerns (APIs, storage, etc.)\n│   ├── api/                # API clients (GoCardless)\n│   ├── storage/            # Storage implementations (AsyncStorage)\n│   └── auth/               # Authentication providers (Firebase)\n├── presentation/           # UI layer\n│   └── hooks/              # Clean architecture React hooks\n├── shared/                 # Shared utilities and constants\n│   ├── constants/          # App constants, colors, config\n│   ├── utils/              # Pure utility functions\n│   └── config/             # Dependency injection container\napp/                        # Expo Router files (existing structure)\n```\n\n## Key Principles Applied\n\n### 1. Dependency Inversion\n- Core business logic doesn't depend on external frameworks\n- Dependencies are injected through interfaces\n- Infrastructure implements core interfaces\n\n### 2. Single Responsibility\n- Each class/module has one reason to change\n- Use cases contain specific business logic\n- Repositories handle data access\n\n### 3. Separation of Concerns\n- Domain entities are pure business objects\n- Use cases orchestrate business operations\n- Infrastructure handles external integrations\n- Presentation handles UI logic\n\n### 4. DRY (Don't Repeat Yourself)\n- Shared utilities and constants\n- Reusable hooks and components\n- Centralized dependency injection\n\n## Migration Steps\n\n### 1. Update Imports\n\nOld way:\n```typescript\nimport { useFirebaseAuth } from '@/hooks/use-firebase-auth';\nimport { useBanks } from '@/hooks/use-banks';\n```\n\nNew way:\n```typescript\nimport { useAuth } from '@/src/presentation/hooks/use-auth.hook';\nimport { useBanks } from '@/src/presentation/hooks/use-banks.hook';\n```\n\n### 2. Use Clean Architecture Hooks\n\nOld way:\n```typescript\nconst { user, login, logout } = useFirebaseAuth();\nconst { banks, loadBanks } = useBanks();\n```\n\nNew way:\n```typescript\nconst { user, login, logout, isLoading, error } = useAuth();\nconst { banks, loadBanks, isLoading, error } = useBanks(user?.uid);\n```\n\n### 3. Update Type Imports\n\nOld way:\n```typescript\nimport { Bank, Transaction } from '@/types';\n```\n\nNew way:\n```typescript\nimport { Bank, Transaction } from '@/src/shared';\n```\n\n### 4. Use Centralized Constants\n\nOld way:\n```typescript\nimport { lightColors, darkColors } from '@/constants/colors';\n```\n\nNew way:\n```typescript\nimport { lightColors, darkColors } from '@/src/shared/constants/colors';\n```\n\n### 5. Use Utility Functions\n\nOld way:\n```typescript\nimport { formatCurrency } from '@/utils/format';\n```\n\nNew way:\n```typescript\nimport { CurrencyUtils } from '@/src/shared/utils/format.utils';\n\n// Usage\nconst formatted = CurrencyUtils.format(amount);\n```\n\n## Benefits of This Architecture\n\n### 1. Testability\n- Business logic is isolated and easily testable\n- Dependencies can be mocked through interfaces\n- Use cases can be tested independently\n\n### 2. Maintainability\n- Clear separation of concerns\n- Changes in one layer don't affect others\n- Easy to locate and modify specific functionality\n\n### 3. Scalability\n- New features follow established patterns\n- Easy to add new data sources or APIs\n- Consistent structure across the application\n\n### 4. Flexibility\n- Easy to swap implementations (e.g., different storage providers)\n- Platform-agnostic business logic\n- Configurable through dependency injection\n\n## Example: Adding a New Feature\n\nTo add a new \"Budget\" feature:\n\n1. **Create Entity**: `src/core/entities/budget.entity.ts`\n2. **Create Repository Interface**: `src/core/repositories/budget.repository.ts`\n3. **Create Use Cases**: `src/core/use-cases/budget.use-cases.ts`\n4. **Implement Repository**: `src/infrastructure/storage/local-budget.repository.ts`\n5. **Add to DI Container**: `src/shared/config/di-container.ts`\n6. **Create Hook**: `src/presentation/hooks/use-budget.hook.ts`\n7. **Use in Components**: Import and use the hook\n\n## File-by-File Migration\n\n### High Priority (Core Functionality)\n1. Update `context/auth-context.tsx` to use new auth hook\n2. Update `hooks/use-banks.ts` to use new banks hook\n3. Update `hooks/use-transactions.ts` to use new transactions hook\n4. Update main screens to use new hooks\n\n### Medium Priority (UI Components)\n1. Update `components/TransactionList.tsx` to use new utilities\n2. Update color imports across components\n3. Update format utility imports\n\n### Low Priority (Configuration)\n1. Update constants imports\n2. Update utility function imports\n3. Clean up old files after migration\n\n## Testing Strategy\n\n### Unit Tests\n- Test use cases with mocked repositories\n- Test utilities with various inputs\n- Test entities for business rules\n\n### Integration Tests\n- Test repository implementations\n- Test API integrations\n- Test hook behavior\n\n### E2E Tests\n- Test complete user flows\n- Test error scenarios\n- Test offline functionality\n\n## Performance Considerations\n\n### Lazy Loading\n- Dependencies are created only when needed\n- Use cases are instantiated on first use\n- Repositories are cached in DI container\n\n### Memory Management\n- Proper cleanup in hooks\n- Avoid memory leaks in subscriptions\n- Efficient data structures\n\n### Caching\n- Repository-level caching\n- Hook-level state management\n- Optimistic updates\n\n## Next Steps\n\n1. **Phase 1**: Implement core architecture (✅ Complete)\n2. **Phase 2**: Migrate existing hooks and contexts\n3. **Phase 3**: Update all components to use new architecture\n4. **Phase 4**: Add comprehensive testing\n5. **Phase 5**: Performance optimization and monitoring\n\n## Resources\n\n- [Clean Architecture by Robert Martin](https://blog.cleancoder.com/uncle-bob/2012/08/13/the-clean-architecture.html)\n- [SOLID Principles](https://en.wikipedia.org/wiki/SOLID)\n- [Dependency Injection Patterns](https://martinfowler.com/articles/injection.html)\n- [React Hooks Best Practices](https://react.dev/learn/reusing-logic-with-custom-hooks)
